I wanted to learn assembly a long time ago. I tried the __asm when it was supported by MSVC x86 before my becoming a programer. The support was dropped when it comes to x64. I thought I would never have the chance to touch assembly again.

Assembly: the code!

However, here I am.

I decided to take it seriouse, building something really works: some simple library about math.

After playing with the MASM for a few moment, I decided to use NASM instead (after reading Understanding Windows x64 Assembly).

Build

I wanted to build a DLL file which I would like to call from my Origin C functions. NASM does not build the DLL for me. Instead, it creates an obj file with the following command:

nasm -f win64 -o f64s_sum.obj f64s f64s_sum.asm
It is the `linker` that creates the DLL file. I open the `x64 Native Tools Command Prompt for VS 2019`, which is fould by searching `x64 Native...` in Windows Start menu, and run the following command: 
link /OUT:f64s.dll f64s_sum.obj /DLL

Bingo! I have the f64s.dll.

The DLL contains no debug info. I should have created another project for debugging. I just cannot wait to see if it works or not. But it is worthy to see if any functions get exported.

The command prompt is open. dumpbin, the COFF/PE Dumper, is capable of that:

dumpbin /EXPORTS f64s.dll
It dumps the following, suggesting the `f64s_sum` is exported and can be called from my Origin C code.

Invoke the Exported Function

Calling functions in a DLL from Origin C is as easy as:

  1. Declare the functions
  2. Invoke the functions

I want to use the functions in my Origin C project. The following is more about programming in OriginPro.

Declare the functions with #pragma dll(...).

The (...), if not empty, tells the compiler where the DLL is located. For example:

  • #pragma dll(E:\repo\f64s\f64s.dll)

    This tells the compiler where the DLL is, explicitly with the full path.

  • #pragma dll(f64s, header)

    You write this code in a header file, telling the compiler that the header file and the DLL file are in the same folder.

  • #pragma dll(kernel32, system)

    This tell the compiler the DLL is in the Windows system folder: %WINDIR%\System32.

  • #pragma dll(zlib, platform)

    This seems to be used internally. It means the DLL is in a particular folder. For 32-bit Origin, the dll is in <OriginEXE>\32Bit, while 64-bit <OriginEXE>\64Bit. OriginLab delivers 64-bit Origin only in recent years and the <OriginEXE>\32Bit folder does not exist any more.

  • #pragma dll()

    It is empty, telling the compiler that it ends declaring functions exported from the DLL.

But wait, it actually does not work, as it always returns 0.0!

Let’s take another look at the definition of f64s_sum

xorpd: Bitwise logical XOR of packed double precision floating-point values.

In case you don’t know what XOR is, here is the definition about Exclusive or (Wikipedia):

Exclusive or or exclusive disjunction is a logical operation that is true if and only if its arguments differ (one is true, the other is false).

What’s the result of A XOR A? ZERO.

The function returns a double value. According to the x64 calling convention:

Non-scalar types including floats, doubles, and vector types such as m128, m128i, __m128d are returned in XMM0.

Right, my code does nothing but returns 0.0. I’ll have another post about how to sum up the values.
