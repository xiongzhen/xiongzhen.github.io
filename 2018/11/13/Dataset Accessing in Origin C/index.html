Dataset Accessing in Origin C

Posted on 2018-11-13

More than mouse clicking and dragging to perform analysis and create graphs in Origin/OriginPro (simply “Origin” hereafter in this post), provided also includes the power of programming, be it LabTalk the scripting language, Origin C the ANSI C closely based programming language with C++ features, or Python the popular and convenient tool. In this post, I would described the most basic operations using Origin C: how to access dataset. It is targeted at whoever has experience in C/C++ programming and is familiar with Origin.

Dataset

Before starting out to write codes, it would be helpful to understand what “dataset” means in Origin. Typically, there are two kinds of datasets: one that you can see directly in a view, either a Worksheet or a Matrix Sheet, which is called associated dataset or dataset for convenience; the other is call loose dataset which is not associated with any view.

You can list all datasets by LabTalk command list s where s is short for series now known as dataset:

list s;

 1 Book1_A     0x0000000006040008  0
*2 Book1_B     0x0000000006040170  0
*3 MBook1      0x00000000060402D8  0
*4 MBook1_2    0x0000000006040440  0
 5 Book1_A@2   0x00000000060405A8  0
*6 Book1_B@2   0x0000000006040710  0
 7 XS          0x00000000060409E0  0
*8 MBook1@2    0x0000000006040B48  0
*9 MBook1_2@2  0x0000000006040CB0  0

Session:
 10            YS   4:{4, 3, 2, 1}

To refer to a particular dataset in Origin C, you can create an instance of Dataset, one of the composite data types supported, with the corresponding name:

Dataset ds("Book1_A")

However, Dataset is not recommended. It does not support string array well. For example, in the above example, if “Book1_A” refers to a worksheet column of text type, ds is invalid. It’s also important to tell if a dataset in session scope or in project scope, to get the worksheet column a dataset belongs to, etc. In practice, the following classes are recommended to access dataset in Origin C:

  1. DataObject
    An instance of DataObject is something you can see directly in your Origin project, for example, a worksheet column and a matrix. You cannot create such instances referring to loose datasets.
    Given a DataObject instance, you can always create an instance of DatasetObject and get its data as vector/matrix. However, the opposite is not guarantteed. Such case we will discuss later.
  2. DatasetObject
    You may take it as an alias of Dataset. However, it supports more operation like masking/unmasking data and provides access to non numeric datasets, etc. It’s always possible to create an DatasetObject instance referring to a dataset, be it loose dataset or not.
    Given a DatasetObject instance, you can get the corresponding DataObject instance only if it’s not a loose dataset. It’s always possible to get its data as vector/matrix.
  3. vector and matrix
    These are classes about composite data type, while the above two are internal Origin objects. Vector and matrix focus more on “data” itself. In other words, mathematical operations.
    Given a vector/matrix, you can create an instance of DatasetObject from it but it’s impossible to find its corresponding instance of DataObject.
    This is reasonable. If a vector/matrix refers to an instance of DataObject, it must be created explicitly. You should pass an DataObject instead of a vector/matrix.

DataObject

Class DataObject is derived from class DataObjectBase, containing only one dataset. Derived from class DataObjectBase as well is class DataPlot, which may have multiple datasets. I’ll write another post to discuss it. Right now, just focus on one dataset.

In most cases, data in a project is saved as either columns in worksheet or matrices in matrix sheet, they are call Column and MatrixObject respectively.

For those who are not familar with the class hierarchy of Origin C, it might be confusing to understand Column or MatrixObject.

  1. Typically, a window in Origin is called a Page. The a spreadsheet with columns are called WorksheetPage while a sheet with matrices are called MatrixPage.
  2. Both WorksheetPage and MatrixPage can have more than one Layer, they are called Worksheet and MatrixLayer respectively.
  3. Each Layer can hold more than one datasets. In Worksheet, it’s called Column holding 1-D dataset. In MatrixLayer, it’s called MatrixObject holding 2-D dataset.

Usually, to create a Column object or a MatrixObject, You should first have the corresponding Layer object. I list most commonly used cases as following:

  1. Given the Layer object
    // Worksheet wks
    Column col(wks, 0);  // Get the first column of wks as col

    

    // MatrixLayer ml;
    MatrixObject mo(ml, 0);  // Get the first matrix object of ml as mo
